Feature Architecture & Interaction Map

This document outlines all the individual features of the AI-Powered Smart Review System and how they seamlessly interact to create a continuous, low-friction feedback loop.

1. The Core Features

Guest-Facing Features (Frontend)

Micro-Incentive Banners: UI prompts that gamify the experience (e.g., "Earn 50 pts to answer one quick question").

Text Review UI: The standard text input area for guests to leave traditional reviews.

"Magic Enhance" Button: An AI tool that takes rough, bulleted user notes and expands them into a polished paragraph.

Dynamic Prompt Questions: 1-2 highly targeted follow-up questions presented to the user based on missing hotel data.

One-Click Tag UI: Binary [Yes/No] or multiple-choice buttons for ultimate low-friction data collection.

Voice Review UI: A conversational audio interface for hands-free reviewing.

AI Semantic Search Bar: A search input that understands intent and context, not just exact keywords.

Backend & AI Processing Features

Data Gap Engine: The logic that queries the database to find what information is stale (e.g., > 6 months old) or missing (e.g., < 3 reviews).

Categorization Pipeline: An LLM function that reads a submitted review and assigns it JSON tags (e.g., ["pool", "staff"]).

Embedding Pipeline: Converts the text of a review into a mathematical vector for searchability.

Async Summarizer: A background task that reads all reviews in a category and generates a single summary sentence, actively mitigating recency bias.

Discrepancy Detector: An AI monitor that compares what a review says against the hotel's official metadata to spot contradictions.

Host-Facing Features (B2B)

Host Dashboard: A dedicated UI for hotel managers or Expedia staff to view flagged property changes.

2. How the Features Interact (The Feedback Loop)

Here is the step-by-step flow of how data moves through the system and triggers different features:

Phase A: The Setup (Before the Review)

A user visits the Hotel Page.

Behind the scenes, the Data Gap Engine queries the Neon Database. It realizes the hotel hasn't had a review about the Gym in 8 months.

The UI renders the Micro-Incentive Banner: "Earn 50 points by telling us about the Gym!"

Phase B: The Review Collection (Text or Voice)

The user decides to leave a review. They have two paths:

Path 1: Text Interaction

The user types sloppy notes into the Text Review UI: "gym was closed, pool cold".

They click the "Magic Enhance" Button. The app sends this to GPT-4o-mini, which replaces the text with: "Unfortunately, the gym was closed during my stay, and I found the pool to be too cold."

Because the user didn't mention the Breakfast (another identified Data Gap), the Dynamic Prompt injects a question below the text box: "Did you happen to try the free breakfast?"

To make it easy, the One-Click Tag UI appears with [Yes, it was good], [Yes, but bad], [No].

Path 2: Voice Interaction

The user clicks the Voice Review UI microphone.

Audio is captured via the browser and sent to OpenAI Whisper (STT) to convert speech to text.

The text is sent to GPT-4o-mini. The LLM is injected with the Data Gap Engine targets. It reads the transcript and generates a conversational response: "Thanks for sharing that the pool was cold. Since you're here, was the gym open?"

The text response is sent to ElevenLabs (TTS), which speaks the question out loud to the user.

Phase C: The Processing Pipeline (Post-Submission)

Once the user hits "Submit" (or finishes the voice chat), the backend fires off a parallel processing pipeline:

Categorization: The review is sent to the Categorization Pipeline, which tags it as ["gym", "pool"] and saves it to the Neon DB.

Vectorization: The review text is sent to OpenAI's embedding model. The resulting vector is saved to the pgvector column in the Neon DB.

Summarization: The new data triggers the Async Summarizer. It pulls all "pool" reviews, creates a fresh 1-sentence summary, and saves it to the DB for the next guest to see.

Discrepancy Check: The Discrepancy Detector notices the user said "gym was closed". It checks the hotel metadata, which says Gym: Open. It immediately creates a flag in the amenity_flags database table.

Phase D: Consumption

Future Guests: When the next guest searches "Are there places to work out?" in the AI Semantic Search Bar, the database uses the vectors from Phase C to instantly return the review about the closed gym.

The Host: The Hotel Manager logs into the Host Dashboard. They see a notification generated by the Discrepancy Detector: "Recent reviews indicate your Gym might be closed. Do you want to update your public listing?"
